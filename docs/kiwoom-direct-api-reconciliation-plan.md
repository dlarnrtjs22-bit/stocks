# 키움 직접응답 API 우선 정합성 계획

## 원칙

- 1순위는 무조건 `키움 앱과 동일한 값을 직접 내려주는 API/TR`을 찾는 것이다.
- `역산`이나 `임의 재계산`은 사용하지 않는다.
- 정말로 직접응답 TR이 없다는 것이 확인된 뒤에만 fallback 계산을 검토한다.

## 현재까지 공식 문서/실호출로 확인한 후보

### 공식 키움 문서에서 확인한 계좌 관련 TR

공식 문서:
- `ka00001` 계좌번호조회
- `ka01690` 일별잔고수익률
- `ka10072` 일자별종목별실현손익요청_일자
- `ka10073` 일자별종목별실현손익요청_기간
- `ka10074` 일자별실현손익요청
- `ka10075` 미체결요청
- `ka10076` 체결요청
- `ka10077` 당일실현손익상세요청
- `ka10085` 계좌수익률요청
- `ka10088` 당일매매일지요청

추가로 현재 프로젝트에서 이미 사용 중인 계좌 TR:
- `kt00018` 계좌평가/잔고 계열
- `kt00005` 계좌평가 현황 계열

근거:
- [키움 REST API 모바일 가이드](https://openapi.kiwoom.com/m/guide/apiguide) 계좌 관련 TR 목록

## 실제 호출로 확인한 상태

### 이미 값이 확인된 TR

- `kt00018`
  - 현재 계좌 평가손익/보유종목 손익을 내려주지만
  - 사용자 키움 앱 숫자와 불일치
- `kt00005`
  - 평가 관련 다른 계좌 수치 제공
  - 역시 현재 앱 숫자와 불일치
- `ka10074`
  - 기간 일자별 실현손익 요약 확인 가능
- `ka10076`
  - 실제 체결내역 조회 가능

### 파라미터 확인이 끝난 TR

- `ka10075` 미체결요청
  - 필수 파라미터:
    - `all_stk_tp`
    - `trde_tp`
    - `stex_tp`
- `ka10076` 체결요청
  - 필수 파라미터:
    - `qry_tp`
    - `sell_tp`
    - `stex_tp`
- `ka10074` 일자별실현손익요청
  - 필수 파라미터:
    - `strt_dt`
    - `end_dt`

## 현재 판단

- 지금 사용자 앱과 차이가 나는 숫자는
  - `장 종료 후 NXT 시세`
  - `키움 내부 평가손익 기준`
  둘 중 하나 또는 둘 다가 반영된 결과일 가능성이 높다.
- 하지만 이 단계에서도 계산으로 맞추면 안 된다.
- 먼저 아래 TR들이 `키움 앱 동일 숫자`를 직접 내려주는지 확인해야 한다.

## 우선 검증할 TR 순서

### 1차 우선순위: 앱 숫자와 가장 직접적으로 연결될 가능성이 높은 TR

1. `ka10085` 계좌수익률요청
2. `ka10088` 당일매매일지요청
3. `ka01690` 일별잔고수익률
4. `ka10077` 당일실현손익상세요청

이 4개는 이름상 키움 앱의
- 종목별 손익
- 계좌 수익률
- 당일 매매일지
와 직접 대응할 가능성이 높다.

### 2차 우선순위: 이미 확인했지만 앱 숫자와 다른 TR

5. `kt00018`
6. `kt00005`

이 둘은 비교 기준용으로만 유지한다.

### 3차 우선순위: 보조/세부 TR

7. `ka10072`
8. `ka10073`
9. `ka10074`
10. `ka10075`
11. `ka10076`

이 그룹은
- 기간 손익
- 체결/미체결
- 일자별 매매 로그
조립용으로 사용한다.

## 검증 방법

### 1. 각 TR의 필수 파라미터 완전 탐색

- 공식 문서 기준으로 파라미터를 확인한다.
- 모르면 직접 호출해서 에러 메시지로 필수 파라미터를 채운다.
- 임의 추측 파라미터는 최소화한다.

### 2. 동일 시점 사용자 앱 숫자와 대조

대조 기준:
- 현대차 평가손익
- 두산에너빌리티 평가손익
- 총 평가손익
- 총 평가금액
- 추정자산

판단 기준:
- 종목별 손익이 앱과 동일하거나 매우 근접한 TR이면 1순위 후보 채택
- 총계만 맞고 종목별이 다르면 보조 TR로 분류

### 3. 장중 / 장후 분리 검증

- 정규장 중
- 장 종료 후
- NXT 시간대

각 구간에서 같은 TR이 어떤 숫자를 주는지 기록한다.

## 승인 후 개발 순서

### 단계 1. 직접응답 TR 탐색 코드만 추가

- 새 브랜치 성격으로
  - 후보 TR 호출
  - 필수 파라미터 정리
  - 응답 구조 비교
  를 먼저 구현/기록

### 단계 2. 직접응답 TR이 확인되면 UI 연결

- `매매내역` 탭 상단/보유종목 숫자를
  해당 TR 값으로 교체
- 이때 기존 역산/실시간 재계산 로직은 제거 또는 비활성화

### 단계 3. 보조 TR 조립

- 체결내역: `ka10076`
- 미체결: `ka10075`
- 기간 손익: `ka10074`, 필요 시 `ka10072/73`

## 역산/재계산에 대한 처리 방침

- 승인 후 개발에서도 `직접응답 TR`이 확인되기 전까지
  손익 숫자를 맞추기 위한 계산 보정은 하지 않는다.
- 만약 모든 공식 후보를 검증한 뒤에도
  앱 숫자와 직접 일치하는 TR이 없을 때만
  그 사실을 별도 문서로 정리하고,
  그 다음 단계에서 fallback 계산 여부를 다시 승인받는다.

## 이번 승인 범위

승인받으면 다음만 수행한다.

1. 공식 문서 기준 후보 TR별 파라미터/응답 구조 검증
2. 실제 계좌 앱 숫자와 TR 응답 비교표 작성
3. 직접응답 TR 선정
4. 선정된 TR 기준으로 매매내역 탭 값 교체
